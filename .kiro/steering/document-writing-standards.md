# 文档编写规范

## 一、核心原则

### 1.1 禁止使用 Emoji 图标
- 文档内容中不要出现任何 Emoji 图标（除非是专门展示 Emoji 的章节）
- 保持文档的专业性和简洁性
- 使用文字描述代替 Emoji 表达

### 1.2 样式规范
- 每次编写文档时，必须参考 `docs/document-template/template.md` 的样式
- 保持文档格式统一、规范、美观
- 遵循 DevManga 主题风格：粗边框、阴影、漫画风格

## 二、文档结构规范

### 2.1 Front Matter（元数据）
每个文档必须包含完整的元数据：

```yaml
---
sidebar_position: 1          # 侧边栏位置（数字）
title: 文档标题              # 页面标题（简洁明了）
description: 文档描述        # SEO 描述（80-160字符）
tags: [标签1, 标签2]         # 文档标签（3-5个）
keywords: [关键词1, 关键词2] # SEO 关键词（5-10个）
authors: [Laby]              # 作者
last_update:                 # 最后更新
  date: 2026-04-24
  author: Laby
---
```

### 2.2 标题层级
- H1 (# 标题)：仅用于文档主标题，每个文档只有一个
- H2 (## 标题)：主要章节，使用中文数字编号（一、二、三）
- H3 (### 标题)：子章节，使用阿拉伯数字编号（1.1、1.2）
- H4-H6：更细的层级，根据需要使用
- 标题要简洁明了，避免过长

### 2.3 文档开头
- 在 Front Matter 后，先写文档标题（H1）
- 然后用 1-2 段简短介绍文档内容
- 使用 `:::tip 核心价值` 提示框总结文档要点

## 三、内容编写规范

### 3.1 文本格式
- **粗体**：用于强调重要概念、关键词
- *斜体*：用于术语、引用、强调
- `行内代码`：用于代码片段、命令、文件名、变量名
- ~~删除线~~：用于标记过时内容
- 链接：使用有意义的链接文本，避免"点击这里"

### 3.2 列表使用
- 无序列表：用于无顺序关系的项目
- 有序列表：用于有步骤或优先级的内容
- 任务列表：用于待办事项或检查清单
- 列表项要简洁，每项一个要点
- 嵌套列表不超过 3 层

### 3.3 代码块规范
- 必须指定语言类型（javascript、python、typescript 等）
- 复杂代码添加标题：`title="文件路径"`
- 需要强调的行使用高亮：`{2,4-6}`
- 长代码块使用行号：`showLineNumbers`
- 代码要完整可运行，添加必要注释
- 避免过长的代码块（超过 50 行考虑拆分）

### 3.4 提示框使用
根据内容性质选择合适的提示框类型：

- `:::note`：一般性说明、补充信息
- `:::tip`：有用的建议、技巧、最佳实践
- `:::info`：额外信息、扩展阅读、背景知识
- `:::warning`：警告、注意事项、潜在问题
- `:::danger`：严重问题、错误、禁止操作
- `:::caution`：小心操作、谨慎使用

提示框使用原则：
- 自定义标题要简洁有力
- 内容要精炼，突出重点
- 可以嵌套代码块、列表、表格
- 不要过度使用，保持文档清爽

### 3.5 表格规范
- 表头使用粗体，内容简洁
- 合理使用对齐方式（左对齐、居中、右对齐）
- 复杂数据使用表格展示，提高可读性
- 表格列数不超过 6 列（移动端考虑）
- 使用 `<br/>` 在单元格内换行

### 3.6 图片和媒体
- 图片必须有 alt 属性（描述性文字）
- 指定合适的宽度（width），保持页面美观
- 图片下方添加说明文字（斜体）
- 图片路径使用 `/img/` 开头的绝对路径
- 图片文件名使用小写字母和连字符

## 四、组件使用规范

### 4.1 Tabs 组件
- 用于展示多种实现方式、多语言对比
- 每个 TabItem 要有清晰的 label
- 可以设置 defaultValue 指定默认选中
- 内容要对等，保持结构一致

### 4.2 卡片组件
- 基础卡片：`<div className="card">`
- 主题色卡片：`card-primary`（蓝色）、`card-success`（绿色）
- 警告卡片：`card-warning`（黄色）、`card-danger`（红色）
- 卡片网格：`<div className="card-grid">` 包裹多个卡片
- 卡片内容要简洁，突出重点

### 4.3 分隔线
- 使用 `---` 分隔不同的主要章节
- 不要过度使用，保持文档流畅
- 重要章节前后使用分隔线

## 五、内容质量规范

### 5.1 准确性
- 技术内容必须准确无误
- 代码示例要经过测试验证
- 版本信息要及时更新
- 引用权威来源，标注出处

### 5.2 完整性
- 提供完整的使用示例
- 包含常见问题和解决方案
- 说明前置条件和依赖
- 提供相关文档链接

### 5.3 可读性
- 使用简洁明了的语言
- 避免过于复杂的句子
- 段落长度适中（3-5 句）
- 使用主动语态
- 技术术语首次出现时解释

### 5.4 实用性
- 提供真实的使用场景
- 包含最佳实践和反模式
- 给出性能优化建议
- 说明注意事项和陷阱

## 六、文件命名规范

### 6.1 文档文件
- 使用小写字母和连字符：`my-document.md`
- 文件名要有意义，反映内容
- 避免使用特殊字符和空格
- 统一使用 `.md` 扩展名（支持 JSX）

### 6.2 目录结构
- 按主题组织文档目录
- 使用 `_category_.json` 配置目录
- 目录名使用小写字母和连字符
- 保持目录层级清晰（不超过 4 层）

## 七、维护和更新

### 7.1 版本管理
- 文档末尾标注最后更新时间
- 重大变更记录版本历史
- 标记过时内容并提供替代方案

### 7.2 一致性检查
- 定期检查文档格式一致性
- 更新过时的代码示例
- 修复失效的链接
- 统一术语使用

### 7.3 反馈改进
- 根据用户反馈改进文档
- 补充常见问题
- 优化难以理解的部分
- 添加更多实例

## 八、特殊注意事项

### 8.1 国际化
- 中文文档使用规范的简体中文
- 英文术语首次出现时标注中文
- 代码注释使用英文或中文（保持一致）
- 考虑多语言版本的维护

### 8.2 SEO 优化
- 标题包含关键词
- description 简洁有力
- 合理使用内部链接
- 图片 alt 属性完整

### 8.3 可访问性
- 图片提供替代文字
- 表格使用语义化标记
- 链接文字有意义
- 颜色不是唯一的信息传达方式
